# BM25

BM25 (Best Match 25) is a probabilistic ranking function that scores documents against a query based on the terms they share. It is the standard baseline for sparse retrieval and has been the dominant ranking function in search engines for over two decades. The "25" refers to it being the 25th iteration in a series of experiments by **Robertson** and **Sparck** Jones at **Okapi research** in the 1990s.

## Intuition

BM25 is best understood as TF-IDF with two specific fixes:

1. **TF saturation**: In TF-IDF, term frequency grows linearly. BM25 applies a
   saturation function so that each additional occurrence of a term contributes less than the previous one. The 10th occurrence of "python" matters much less than the 1st.

2. **Document length normalization**: Longer documents naturally contain more terms and accumulate higher TF scores. BM25 explicitly penalizes documents that are longer than average, so a term appearing 3 times in a short document scores higher than the same term appearing 3 times in a very long one.

These two fixes make BM25 significantly more accurate than raw TF-IDF in practice.

## The Formula

For a query q with terms t₁, t₂, ..., tₙ and a document d:

```bash
BM25(d, q) = Σ IDF(tᵢ) × [ TF(tᵢ, d) × (k₁ + 1) ]
                        [ ────────────────────────────────────────── ]
                        [ TF(tᵢ, d) + k₁ × (1 - b + b × |d|/avgdl) ]
```

Where:

- TF(tᵢ, d) = frequency of term tᵢ in document d
- |d| = length of document d (number of terms)
- avgdl = average document length across the corpus
- k₁ = TF saturation parameter (typically 1.2 to 2.0)
- b = length normalization parameter (typically 0.75)
- IDF(tᵢ) = inverse document frequency of term tᵢ

IDF in BM25 (Robertson-Sparck Jones variant):

```bash
IDF(t) = log( (N - df(t) + 0.5) / (df(t) + 0.5) + 1 )
```

Where N = total documents, df(t) = documents containing term t.

## Understanding the Parameters

### k₁ - TF saturation

k₁ controls how quickly additional occurrences of a term stop mattering.

```bash
tf_component = TF × (k₁ + 1) / (TF + k₁)
```

As TF → ∞, this component → (k₁ + 1). It never exceeds k₁ + 1 regardless of
how many times the term appears. This is the saturation ceiling.

```bash
k₁ = 0  → TF is ignored entirely (pure IDF)
k₁ = ∞  → recovers linear TF growth (like raw TF-IDF)
k₁ = 1.2 → fast saturation, typical for short documents
k₁ = 2.0 → slower saturation, typical for longer documents
```

Visualizing saturation with k₁ = 1.5:

```bash
TF:    1     2     3     5     10    100
score: 0.83  1.20  1.40  1.56  1.71  1.96  (ceiling = 2.5)
```

Each additional occurrence contributes less. The gap between TF=1 and TF=2 is large;
the gap between TF=10 and TF=100 is tiny.

### b — length normalization

b controls how much document length affects the score.

```bash
length_norm = 1 - b + b × (|d| / avgdl)
```

```bash
b = 0  → no length normalization (document length ignored)
b = 1  → full length normalization (score fully adjusted for length)
b = 0.75 → standard choice, partial normalization
```

When |d| = avgdl, length_norm = 1 — no adjustment.
When |d| > avgdl, length_norm > 1 — TF is penalized (long doc).
When |d| < avgdl, length_norm < 1 — TF is boosted (short doc).

## Worked Example

Corpus:

```bash
D1: "python search index python"      (4 terms)
D2: "python tutorial beginners"       (3 terms)
D3: "search engine design search"     (4 terms)
D4: "cooking recipes dinner"          (3 terms)
```

N = 4, avgdl = (4+3+4+3)/4 = 3.5
k₁ = 1.5, b = 0.75

Query: "python search"

### Score for D1

**Term "python":**

```bash
TF("python", D1)  = 2
df("python")      = 2  (D1, D2)
IDF("python")     = log((4 - 2 + 0.5) / (2 + 0.5) + 1) = log(1 + 1) = 0.693

length_norm = 1 - 0.75 + 0.75 × (4/3.5) = 0.25 + 0.857 = 1.107

tf_component = 2 × (1.5 + 1) / (2 + 1.5 × 1.107)
             = 2 × 2.5 / (2 + 1.661)
             = 5 / 3.661
             = 1.366

score("python", D1) = 0.693 × 1.366 = 0.946
```

**Term "search":**

```bash
TF("search", D1)  = 1
df("search")      = 2  (D1, D3)
IDF("search")     = log((4 - 2 + 0.5) / (2 + 0.5) + 1) = 0.693

tf_component = 1 × 2.5 / (1 + 1.5 × 1.107)
             = 2.5 / (1 + 1.661)
             = 2.5 / 2.661
             = 0.939

score("search", D1) = 0.693 × 0.939 = 0.651
```

**Total BM25(D1, q) = 0.946 + 0.651 = 1.597**

### Score for D2

**Term "python":**

```bash
TF("python", D2)  = 1
length_norm = 1 - 0.75 + 0.75 × (3/3.5) = 0.25 + 0.643 = 0.893

tf_component = 1 × 2.5 / (1 + 1.5 × 0.893)
             = 2.5 / 2.339
             = 1.069

score("python", D2) = 0.693 × 1.069 = 0.741
```

**Term "search":** TF = 0 → score = 0

**Total BM25(D2, q) = 0.741**

Ranking: D1 (1.597) > D2 (0.741) > D3 (scores only on "search") > D4 (0)

D1 ranks highest — it contains both query terms, "python" appears twice, and the
document is close to average length. D4 scores zero because neither query term
appears in it.

## BM25 vs TF-IDF

| Property                 | TF-IDF               | BM25                        |
| ------------------------ | -------------------- | --------------------------- |
| TF growth                | Linear, unbounded    | Saturating, bounded by k₁+1 |
| Length normalization     | Via cosine (partial) | Explicit via b and avgdl    |
| Parameters               | None                 | k₁ and b (tunable)          |
| IDF formula              | log(N/df)            | Robertson-Sparck Jones IDF  |
| Ranking quality          | Good baseline        | Better baseline             |
| Still used in production | Less common          | Widely used (Elasticsearch, |
|                          |                      | Lucene, Solr, BM25s)        |

## Variants of BM25

| Variant   | Difference from standard BM25                                  |
| --------- | -------------------------------------------------------------- |
| BM25+     | Adds a floor to TF component so zero-TF terms still contribute |
| BM25L     | Alternative length normalization, better for long documents    |
| BM25-adpt | Adapts k₁ per query term based on corpus statistics            |
| BM25F     | Extends BM25 to structured documents with multiple fields      |

BM25F is particularly useful for documents with title, body, and anchor text fields —
each field gets its own k₁ and b parameters.

## Where BM25 still lives today

BM25 is not just a historical model, it is actively used in production:

- **Elasticsearch / OpenSearch**: default ranking function
- **Lucene / Solr**: core ranking model
- **Hybrid search pipelines**: BM25 as the sparse retrieval component alongside
  dense neural retrieval
- **Baseline**: every neural IR paper reports BM25 as the comparison baseline

In hybrid search, BM25 and dense retrieval are complementary. BM25 is strong on
exact keyword matches and rare terms; dense retrieval is strong on semantic
similarity and paraphrase. Combining both consistently outperforms either alone.

## Where this fits in the progression

```bash
Boolean model       → binary match, no ranking
Extended Boolean    → soft match, partial scoring
Vector Space Model  → geometric similarity, continuous ranking
TF-IDF              → term weighting scheme for VSM
BM25                → fixes TF saturation and length normalization  ← you are here
Neural IR           → replaces sparse vectors with dense embeddings
```

BM25 is the end of the classical IR story. Everything after it; dense retrieval,
bi-encoders, cross-encoders; is about replacing hand-crafted term statistics with learned representations. But BM25 remains the baseline that all of them are measured against.

## My Summary

BM25 improves on TF-IDF by introducing two fixes: a saturation function that prevents high term frequencies from dominating the score, and explicit document length normalization that penalizes long documents relative to the corpus average. Two parameters: k₁ for saturation speed and b for length normalization strength: make it tunable for different corpora. BM25 has been the dominant sparse retrieval baseline for over two decades and remains in active production use in Elasticsearch, Lucene, and hybrid search pipelines. It is the benchmark every neural retrieval model is compared against.
